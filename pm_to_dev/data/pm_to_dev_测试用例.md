
# 产品视角的需求到开发工程师视角的技术点的转换的测试用例

[← 返回项目主页](../../README.md) | [pm_to_dev 项目文档](../README.md)

## 需求

```
我们需要一个智能推荐功能,提升用户停留时长
```

## 逐步澄清需求内容

```
这个智能推荐功能要解决什么具体的业务痛点或目标？提升用户活跃度10%     
用户场景：针对vip用户
用户会在什么具体场景下使用这个功能？用户首页后，banner中显示商品信息
第一期最核心的推荐能力是什么？商品推荐
是否有必须包含的功能/特性？不需要
期望的上线时间和是否有硬性截止日期？期望三天内上线

除了提升VIP用户活跃度10%，还有其他量化的业务指标吗？如点击率、转化率等？没有了！
如何定义“活跃度”？用户每日打开app
VIP用户指什么标准？付费用户
用户如果对推荐商品不感兴趣，有无期望的反馈/操作？不需要统计
同一位用户每次访问首页Banner，商品是否需要做去重或变化？要去重
推荐商品的来源范围是否有限制？无限制
推荐展示数量是多少？1万到10万
时间规划：三天内上线是否包含测试验收时间
该功能与现有上线计划有无冲突或优先级变动？最高优先级


推荐商品的『去重』具体指什么？同一个用户每天不重复
用户每日打开APP指的是每个自然日只推荐一次吗？只要打开app就推荐，固定的首页banner
推荐商品的数量：1万到10万是指总共商品池规模
Banner中的推荐商品：仅静态展示一条
三天内上线包含业务侧/产品侧的测试验收、bug修复、上线运营等完整流程
推荐商品池完全无限制，是否有确定不推荐的商品类型或黑名单？不需要考虑这些
推荐商品需多长时间更换一次？推荐后固定不变
如用户未登录或非VIP，Banner是否需要隐藏或展示其它内容？其他用户都不展示banner


1）当前VIP用户的日活跃用户（DAU）大致是多少？ 用户有10万，20%用户有日活
2）商品推荐的池子规模是1万-10万，请问当前是否已经有可以直接用作推荐的数据以及商品库？已经准备好了商品
3）是否需要考虑冷启动的用户（即首次成为VIP没有行为记录的情况）？不考虑
4）Banner只展示一条商品，推荐商品每天不重复，若商品被用户看到后，是否要记录曝光避免再次推荐？不记录
5）对于三天内上线，是否允许后续持续优化，第一期只上线最基础的规则推荐满足上线目标？ 不需要优化，一次性完成
```
